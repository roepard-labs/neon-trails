Descripción de casos de uso

Parte textual del modelo de casos de uso (curso): flujo normal paso a paso (camino feliz), flujos alternos cuando cambia la condición y excepciones ante fallos o datos inválidos. Las fichas siguen la idea del documento de clase y se anclan al código real (Laravel + Vue).

Plantilla guía: nombre del caso, actor(es), tipo primario/secundario, precondiciones, descripción, flujo normal de eventos (paso numerado con interacción actor–sistema), flujos alternos, postcondiciones y excepciones etiquetadas (p. ej. EX-01).

Referencias en el código

  • OpenAPI / Scramble: UI GET /docs/api, JSON GET /docs/api.json; Route::redirect('/docs', '/docs/api') en routes/web.php; alias bajo API /api/documentation → /docs/api en routes/api.php.
  • Vue Router: task-manager-web/src/router/index.js — guest.home, tasks.index (requiresAuth), auth.verify-email; guarda en beforeEach usando token y usuario en localStorage.

CU-DOC-00 — Consultar documentación de la API (Scramble) Secundario

Actor principal
Desarrollador o usuario técnico
Precondiciones
Conocer la URL base de la API (APP_URL o VITE_API_URL). Acceso sujeto a SCRAMBLE_PUBLIC_DOCS / SCRAMBLE_ALLOWED_EMAILS.
Flujo principal
1) En guest.home, pulsar “See API documentation”. 2) El navegador abre VITE_API_URL/docs. 3) Laravel redirige a /docs/api (Scramble). 4) Opcional: descargar contrato en /docs/api.json.
Flujos alternos
Docs privadas: el usuario debe autenticarse en la UI de Scramble según política del servidor.
Postcondiciones
Contrato OpenAPI 3 visible o serializable para integración.
Excepciones
  • EX-01: dominio API mal configurado en .env → página en blanco o CORS; revisar VITE_API_URL.

CU-REG-01 — Registrarse Primario

Actor principal
Invitado
Precondiciones
No posee sesión válida; el correo no está registrado (o el backend devuelve error de validación).
Flujo principal
1) Solicita registro en la SPA (ruta guest.home). 2) POST /api/register. 3) Token y usuario vía store Pinia. 4) Persistencia en localStorage. 5) Navegación a auth.verify-email o tasks.index según email_verified (ver HomeView / auth store).
Flujos alternos
A) Token ausente pero usuario creado: el cliente puede reintentar login. B) Error 422: muestra mensajes de validación.
Postcondiciones
Cuenta creada; sesión iniciada si el token fue emitido; usuario almacenado en estado local.
Reglas de negocio
Contraseña confirmada; políticas de validación definidas en el Form Request del backend.

CU-AUTH-02 — Iniciar sesión Primario

Descripción: el invitado se autentica contra la API y obtiene un token Sanctum para activar los casos de uso que incluyen validación de sesión (gestión de tareas).

Actor(es)
Invitado
Precondiciones
Existe una cuenta registrada; la SPA está cargada y el usuario puede acceder al modal de login (AuthModal.vue).

Flujo normal de eventos

Paso Actor Sistema
1 En la ruta guest.home, abre login y envía email y contraseña. Muestra formulario y, al enviar, ejecuta POST /api/login con Axios.
2 — Laravel valida credenciales; responde con token y usuario (Sanctum).
3 — El store Pinia guarda token y usuario; persiste en localStorage.
4 — Vue Router navega a tasks.index si el correo está verificado; si no, a auth.verify-email (según HomeView / auth store).
Flujos alternos
FA-1 (paso 1–2): credenciales incorrectas → la API responde error; la UI muestra mensaje y no guarda token (permanece en flujo de login).
Postcondiciones
Sesión iniciada: token enviado en Authorization: Bearer en llamadas posteriores; rutas con requiresAuth accesibles.
Excepciones
  • EX-01 (paso 2): respuesta sin token cuando se esperaba → el store lanza error controlado (“Authentication token missing”).
  • EX-02 (paso 2): error de red → mensaje en UI; no se asume sesión válida.
Reglas / notas
router.beforeEach impide entrar a /tasks sin token; coincide con el caso incluido “validar sesión” del diagrama.

CU-VER-03 — Verificar correo electrónico Secundario

Actor principal
Usuario autenticado
Precondiciones
Sesión activa; email marcado como no verificado en el modelo de usuario.
Flujo principal
1) Vista auth.verify-email (/verify-email). 2) POST /api/email/verification-notification. 3) Enlace firmado GET /api/email/verify/{id}/{hash}. 4) refreshCurrentUser en store auth.
Flujos alternos
Enlace expirado o inválido: mensaje de error; posible nuevo envío sujeto a throttle.
Postcondiciones
Campo email_verified coherente en cliente y servidor.
Reglas de negocio
Rutas firmadas y límites de frecuencia en el endpoint de notificación.

CU-TASK-04 — Listar tareas (paginado) Primario

Actor principal
Usuario autenticado
Precondiciones
Token válido; permisos de acceso a la colección de tareas del usuario.
Flujo principal
1) Navega a tasks.index (/tasks). 2) GET /api/tasks?page=n. 3) Store Pinia + vista lista.
Flujos alternos
Página vacía: la colección llega sin ítems; la UI muestra lista vacía (no es error).
Postcondiciones
Estado Pinia actualizado con tareas y página actual.
Excepciones
  • EX-01: HTTP 401 → sesión inválida; conviene limpiar token y volver a guest.home.
  • EX-02: error de red o 5xx → mensaje de error en store; lista puede quedar desactualizada hasta nuevo intento.
Reglas de negocio
Solo se listan tareas autorizadas por políticas del backend. Este caso incluye validación de token (Sanctum).

CU-TASK-05 — Crear / editar / eliminar tarea

Actor principal
Usuario autenticado
Precondiciones
Sesión válida; formulario completo según reglas (título, estado, fecha, etc.).
Flujo principal
Crear: POST /api/tasks. Editar: PUT/PATCH /api/tasks/{id}. Eliminar: DELETE /api/tasks/{id}. Tras respuesta OK, actualiza lista local y escucha eventos realtime.
Flujos alternos
Validación fallida o 404 en recurso: mensaje en UI sin corromper lista.
Postcondiciones
Recurso persistido o eliminado; vista coherente con el servidor.
Reglas de negocio
Los eventos broadcast complementan pero no sustituyen la confirmación HTTP.

CU-RT-06 — Sincronizar en tiempo real

Actor principal
Usuario autenticado (y sistema de broadcasting)
Precondiciones
Echo inicializado; para canales privados, POST /api/broadcasting/auth autorizado.
Flujo principal
1) Suscripción a tasks y user.{id}.tasks. 2) Recepción de TaskCreated, TaskUpdated, TaskDeleted. 3) Actualización optimista del store.
Flujos alternos
Reverb no disponible: CRUD por HTTP sigue operativo; la UI puede mostrar estado degradado.
Postcondiciones
Listado refleja cambios externos sin recarga manual.
Reglas de negocio
Los payloads deben mapear a los mismos identificadores que la API REST.

CU-AUTH-07 — Cerrar sesión

Actor principal
Usuario autenticado
Precondiciones
Token almacenado localmente.
Flujo principal
1) Solicita logout. 2) POST /api/logout (best-effort). 3) Limpia token y usuario en cliente.
Flujos alternos
Error de red: igualmente se limpia la sesión local para evitar estado inconsistente.
Postcondiciones
No se envían peticiones autenticadas hasta un nuevo login.
Reglas de negocio
Desconexión de canales Echo al salir de la vista de tareas (implementación en store).

Cierre

Este conjunto de casos de uso cubre el núcleo del sistema TaskManager y conecta la exposición académica con decisiones de implementación reales (Sanctum, rutas API, Echo/Reverb). Para despliegue estático, sirve la carpeta web/ en la ruta /web de tu reverse proxy o plataforma.
