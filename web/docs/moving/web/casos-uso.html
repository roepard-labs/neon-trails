Diagrama de casos de uso

Modelo en línea con la materia de requerimientos: actores fuera del rectángulo del sistema, casos de uso primarios (objetivo central del software) y secundarios (apoyo), relaciones «include» (siempre se ejecuta el incluido) y mensajes en tiempo real.

Elementos del diagrama (UML — curso)

  • Sistema: rectángulo que agrupa los casos de uso del software TaskManager.
  • Actores: quien interactúa con el sistema (personas u otros sistemas).
  • Casos de uso primarios: registrar/iniciar sesión, gestionar tareas (listar, crear, editar, borrar), cerrar sesión.
  • Casos de uso secundarios: consultar documentación OpenAPI (Scramble), verificar correo, sincronización en tiempo real vía Reverb/Echo.
  • «Include»: operaciones sobre tareas siempre requieren validar sesión/token (Sanctum + guards). En el diagrama: flecha include hacia el caso reutilizado.
  • Extend (opcional): aquí no se grafica para no saturar; ejemplos de flujo opcional serían “activar notificaciones” o pasos condicionales ya cubiertos en la descripción textual.

Invitado

Casos de acceso inicial y consulta de contrato (RF usuario). Entrada típica: guest.home; docs en VITE_API_URL/docs → Scramble /docs/api.

Usuario autenticado

Ejecuta los RF de negocio sobre tareas y sesión; la verificación de correo puede ser condición previa según política (RNF seguridad / flujo de cuenta).

Tiempo real

Caso de soporte secundario: Echo + Reverb; broadcast auth con el mismo token Sanctum (RF sistema + RNF coherencia en vivo).
